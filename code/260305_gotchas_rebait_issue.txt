GOTCHA: GraphMazeEnv Rebait Order-of-Operations Issue
====================================================
Fixed: 2026-03-05, commit a2e9c93
Affects: ALL simulation CSVs in data_out/rl_sims/ generated before this date

WHAT WAS WRONG
--------------
GraphMazeEnv.step() had the wrong order of operations for reward rebait:

  OLD (buggy):
    1. rwd_per_node[current] = 0               # REMOVE consumed state
    2. if sum(rwd_per_node) < min_allowed:      # CHECK rebait threshold
    3.     rwd_per_node = deepcopy(initial)      # REBAIT (restores consumed!)

  CORRECT (RPi experiment, line 559 of 240829_latMaz_exp-mop0half-STABLE.py):
    1. if len(rewarded_states) == thresh + 1:   # CHECK rebait threshold
    2.     rewarded_states = deepcopy(INITIAL)   # REBAIT
    3. rewarded_states.remove(state_t)           # THEN REMOVE

  FIXED GraphMazeEnv:
    1. if sum(rwd_per_node) == min_allowed + 1:  # CHECK before removal
    2.     rwd_per_node = deepcopy(initial)       # REBAIT
    3. rwd_per_node[current] = 0                  # THEN REMOVE

Two issues in the old code:
  a) Removing before checking shifts the threshold comparison by one
  b) Rebaiting after removal means deepcopy(initial) restores the consumed state

WHY WE KNOW THE FIX IS CORRECT
-------------------------------
The RPi code IS ground truth -- it ran during the actual mouse experiments.
The yoking DF records the actual n_rewards each mouse earned.

Simulating both logics on 185 validated mouse trajectories:
  - RPi (fixed) logic matches actual n_rewards: 181/185 (97.8%)
  - Old GraphMazeEnv logic matches:              122/185 (65.9%)
  - 63 sessions (34.1%) produce different reward counts

The fix generally INCREASES reward counts (mean delta +6), because the old
logic under-counted by removing the consumed state before checking rebait.
